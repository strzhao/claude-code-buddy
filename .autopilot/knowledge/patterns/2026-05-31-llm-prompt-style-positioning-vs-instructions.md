# [2026-05-31] LLM systemPrompt：**风格定位词** > **扩展指令** > 模板填空

## 实测对比（5 个翻译 case dry-run）

| systemPrompt 策略 | 平均输出 | 质量 |
|---|---:|---|
| **A** "对学习者最有用的翻译/辨析/文化背景" + 自由发挥 + action 标签 | 2227 字 | ❌ 像教材，超出用户单次查词的认知负担 |
| **X** 极简："翻译用户输入。给英文加 🔊...给译义加 📋..." | 73 字 | ❌ 模型把 action 协议当**主任务**，输出 1 译义就停 |
| **Y** 无 systemPrompt，user 包装"翻译 X。给英文加..." | 100 字 | ❌ 同 X，且有时直接把指令字面输出 |
| **Z** "**英汉/汉英查词助手**。输出风格像字典，先给最常用译义，每个配一个简短例句，不写文化背景/辨析。" + action 标签 | **277 字** | ✅ **接近有道词典 — 3 译义 + 例句 + action 标签自然贴在英文/译义上** |
| **B 基准** 无 systemPrompt，user="翻译 buddy" | 311 字 | ✅ 模型默认查词风格，干净 |

## 结论：三层 prompt 哲学

### 1. 扩展指令是反模式（A 错误）
"对学习者最有用 / 内容自由发挥 / 输出文化背景" 这类**目标性指令**触发模型的"展示型 mode"——它理解为"让我证明我懂得多" → 写 2000+ 字教材。**用户单次查词意图被完全淹没**。

### 2. 极简协议告知也是反模式（X/Y 错误）
当 systemPrompt 只剩"action 协议告知"时，模型把"贴标签"理解为**核心任务**，把翻译降级为附属，给 1 个译义就停（"任务完成"）。Y 甚至直接把指令文本回显。**协议指令的权重被极简上下文放大**。

### 3. 风格定位 = 正解（Z 成功）
用**已知 schema 的角色名**（"英汉/汉英查词助手"、"摘要助手"、"代码评审员"）作为定位，模型自动激活对应输出 schema（查词 = 译义+例句、摘要 = bullet list、评审 = 问题+建议）。配 action 标签作为格式要求（不是主任务），出来就对了。

## How to apply

写 systemPrompt 的优先级：
1. **第 1 句**：用一个**已知角色名定位**（"X 助手"、"X 顾问"），让模型激活对应 schema。**不要写"对 X 最有用"或"自由发挥"这类目标指令**。
2. **第 2 句**：用**形态约束词**（"像字典"、"像 changelog"、"像 git commit message"）补充输出形态，**不写数字**（"3 个译义" 死板），让 AI 自己判断密度。
3. **第 3 句**：**negative 约束**（"不写文化背景/不写辨析/不写对话"）剪掉不想要的扩展。
4. **末句**：**协议/格式要求**（"给英文加 🔊"）作为辅助，不要让它单独成为主任务。

## 配套实例（translate plugin v0.5.0）
```
你是英汉/汉英查词助手。输出风格像字典：先给最常用的译义，每个译义配一个简短例句，不写文化背景、不写用法辨析、不延伸对话。给所有英文（含例句中的英文）加 <action:speak text="英文">🔊</action> 朗读，给译义加 <action:copy text="译文">📋</action> 复制。
```
157 字（vs A 版 1273 字 / 87% 减少），平均输出 277 字（vs A 版 2227 字 / 87% 减少 + 质量更好）。

## 反例避雷（与本知识库以前条目修订）
- ❌ ~~5-few-shot 真实示例驱动~~（之前 [[2026-05-31-llm-fewshot-vs-template-placeholders]] 的结论）— 也是反模式，**示例会把模型默认能力压成"填空机"**，损失更多。修订后 patten 在 Z 风格定位 + 简短例句要求即可，**不需要给完整 few-shot**。
