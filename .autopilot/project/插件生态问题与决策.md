# 插件生态系统性问题与决策（单独任务设计输入）

> 本文档不是当前实施方案，而是**单独任务的设计输入**：梳理社区插件化暴露的系统性问题 + 已达成的关键决策 + 方案 3（依赖一键安装）的深化设计要点。当前「社区插件 git 化」改造先收尾（见末节），qr shell 化 + 依赖机制进入单独任务。

## 背景

上一轮「社区插件 git 化」完成了：官方插件剥离独立 git monorepo（`strzhao/buddy-official-plugins`）+ app 编译时 build-time fetch + 插件页自动更新开关 + TOFU 严格首次。但在 QA 真机验收时暴露了**编译型插件与社区理念的结构性冲突**，进而牵出外部依赖管理、依赖声明、分发模式等一系列系统性问题。本文档把这些问题与决策固化，供单独任务实施。

## 系统性问题清单

### P1. 编译型插件热更新在用户侧失效（已定位，qr 是当前唯一实例）
- **现象**：qr 用 Swift（`qr-gen.swift` → universal binary via CoreImage，零外部依赖）。源码在 monorepo，binary 不入库（`.gitignore` 排除）。app 构建时 `Makefile build-qr-gen` 编译进 bundle；但**用户侧 release app 热更新**（`MarketplaceManager.syncFromRemote` → `installPlugin` → `PluginSourceResolver.gitSubdir` git clone）只拉源码，无编译环境 → qr-gen binary 缺失 → `cmd ./qr-gen` 找不到 → `LauncherError.pluginCrash`。
- **关键修正**：问题不是权限丢失（`ensureStdinChmod` `MarketplaceManager:621,753-777` 已兜底 chmod 0o755），是**文件缺失**。
- **泛化**：未来任何编译型插件（Go/Rust/Swift）都重蹈此问题。编译型与「源码即分发 + 热更新拉源码即可用」的社区理念冲突。

### P2. 外部依赖管理机制缺失
- 插件可声明 `requiredPath`（命令存在性检查），但 app **缺少「检测缺失 → 引导/安装」的统一机制**。
- 用户侧装外部依赖（如 qrencode）的体验未定义：报错？引导？一键装？

### P3. 依赖声明能力弱
- `plugin.json` 只有 `requiredPath`（命令名存在性），**无 brew/apt 依赖映射、无版本约束、无安装引导元数据**。
- 框架无法从 `requiredPath: ["qrencode"]` 推断「用 brew install qrencode 装」。

### P4. 插件分发模式未统一
- 脚本型（shell/python，源码入库）vs 编译型（binary 分发）混用，框架对编译型分发无规范（入库 binary / release asset / 运行时编译三选一未定）。

### P5. 安装体验：社区插件的「最后一公里」
- 用户装一个依赖外部工具的插件（如 qr→qrencode），体验割裂（装插件 ≠ 装依赖）。需统一「插件 + 依赖」的一体化安装体验。

## 已达成的关键决策

### D1. qr 改 shell + qrencode（从源头消除编译型）
- **方向**：qr 从 Swift 编译型 → shell 脚本调 `qrencode`（脚本型，社区友好）。
- **理由**：脚本型是社区插件的天然默认（开发者门槛低、源码即分发、热更新拉源码即可用、彻底消除 P1 这一类问题）。qr 作为官方示范插件，示范「脚本型社区插件正确姿势」比示范编译型更有教育意义。
- **代价**：qr 从零依赖（CoreImage 系统自带）→ qrencode 外部依赖（用户需装一次）。引出 P2/P3/P5。
- **`qr-gen.sh` 契约**（对齐 Swift 版）：command mode，读用户输入 text（args 或 stdin，实现时确认 `StdinExecutor` 传递机制）→ `qrencode -o "$BUDDY_OUTPUT_IMAGE" "$text"` 生成 PNG → stdout 简短回显。
- **plugin.json 改造**：`cmd: ./qr-gen.sh` + `requiredPath: ["qrencode"]`（+ 依赖声明扩展，见 D3）。
- **Makefile 改造**：移除 `build-qr-gen` target + 调整链式依赖（`fetch-plugins → fix-plugin-perms → build/bundle`，不再编译 qr）。
- **monorepo plugins/qr/ 改造**：删 `qr-gen.swift`，加 `qr-gen.sh`，plugin.json 改 cmd + requiredPath，README 说明依赖。

### D2. qrencode 依赖采用方案 3（一键安装按钮），需好好设计
- **否决方案 1（脚本内 stderr 提示）**：不醒目，用户易忽略。
- **否决方案 2（requiredPath + NSAlert 引导复制命令）**：体验中，用户仍需手动终端装。
- **选定方案 3**：app 检测依赖缺失 → NSAlert 确认 → 用户点「安装」→ app 调 `brew install <dep>`。体验最好，但需精心设计安全边界（见下）。
- **接受的红线调整**：app 自动装系统包通常禁止，但**用户在 NSAlert 明确授权 + 显示完整命令 + 单次操作 + 无 sudo**（brew 不需 sudo）的前提下，可接受。这与其他 macOS 工具（如 Homebrew app、某些 IDE 装工具链）一致。

### D3. plugin.json 依赖声明扩展（支撑方案 3）
- `requiredPath` 保留（命令存在性检查，通用）。
- 新增 `deps`（或扩展 requiredPath 为对象）：声明安装方式映射，如 `deps: [{"check": "qrencode", "brew": "qrencode"}]`。app 据此知道「缺失 qrencode → brew install qrencode」。
- 设计要点：brew 为主，预留多包管理器（MacPorts/源码）扩展位；版本约束 YAGNI（暂不做）。

## 方案 3 深化设计要点（单独任务核心）

### 触发
- 插件执行前（`PluginDispatcher`/`StdinExecutor` 入口）或安装时（`installPlugin` 后）检查 `requiredPath`/`deps`。
- 命令缺失 → 触发安装流程。

### UI（NSAlert 确认）
- 标题：「{plugin} 需要 {dep}」
- 正文：「该插件依赖 `{dep}`（当前未安装）。将通过 Homebrew 安装：`brew install {dep}`」
- 按钮：「安装」/「取消」
- **安全要素**：显示完整命令（透明）、用户主动点「安装」（授权）、不静默、单次。

### 执行
- 用户点「安装」→ app 起 `Process` 跑 `brew install {dep}`（子进程）。
- 进度：实时输出（NSAlert 附带进度区 或 单独安装窗口）。
- 完成：重新检查 `requiredPath` → 成功则继续执行插件；失败则错误提示。

### 安全与边界（重点）
- **brew 缺失检测**：先 `command -v brew`；缺失则引导装 Homebrew（brew.sh 链接，不自动装 brew——装 brew 是更大副作用）。
- **无 sudo**：brew install 不需 sudo；若触发 sudo 提示视为异常，中止。
- **失败处理**：brew install 失败（网络/已存在冲突/权限）→ 显示 stderr + 建议，不卡死。
- **超时**：brew install 给合理超时（网络慢），可取消。
- **审计日志**：`BuddyLogger` 记录每次依赖安装（dep、命令、结果），subsystem `plugin`。
- **用户控制**：设置页可关「自动依赖安装」（全局开关，默认开）；关时回退到方案 2 行为（显示命令 + 复制，不自动装）。

### 通用化
- 不只 qrencode：框架统一的 plugin deps 安装机制。任何插件声明 `deps` 都走此流程。
- 预留：多包管理器（brew 主，MacPorts/源码备）、版本约束（YAGNI 暂不做）。
