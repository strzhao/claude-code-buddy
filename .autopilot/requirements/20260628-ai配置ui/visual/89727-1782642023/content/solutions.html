候选方案对比

方案 A · 推荐
最小切入 — 直接复用基础设施
架构改动
  • SettingsSection 加 .ai case（1 行）
  • 新建 ProviderSettingsViewController（~250 行）
  • 注册到 SettingsSplitViewController（1 行）
表单实现
  • 直接用 AppKit 原生控件：NSPopUpButton（kind）、NSTextField（模型/URL）、NSSecureTextField（密钥）
  • 布局用现有 SettingsGroupView + SettingsGroupLabel
  • 不新建通用表单组件
连接测试
  • VC 内直接通过 ProviderFactory.create() 构造 provider，调用一个轻量 chat completion（单条 "ping" 消息）
  • 解析 HTTP 状态码 / error body 映射为中文错误提示
  • 成功展示延迟 + 模型确认
数据流
  • 读写 LauncherConfig.load()/save()（JSON）
  • API key 走 SecretStoreFactory.create()
  • 工具状态从 MetaTools.all 读取展示
  • 提示词从 DefaultAgentPrompt.system 读取展示
改动面小 快速交付 测试简单
方案 B
组件化 — 先建 SettingsFormField 组件库
与方案 A 相同
  • SettingsSection.ai + ProviderSettingsViewController
  • 连接测试逻辑、数据流不变
差异
  • 先建 3 个可复用组件：SettingsTextField、SettingsSecureField、SettingsDropdown
  • 每个组件封装 label + 副标题 + 输入控件的统一布局
  • ProviderSettingsViewController 通过组合这些组件构建 UI
  • 约定：新增表单组件需走 Settings/Components/ 目录、单测
可复用 一致性好 YAGNI 风险 工期 ×1.5
推荐方案 A
理由：(1) 当前仅「AI 配置」一个页面需要表单输入，建通用组件库是 YAGNI； (2) 原生 AppKit 控件足够用，不需要封装层； (3) 如果后续其他设置页也需要表单字段，届时再提取组件成本极低（抽象 ex post > 预判 ex ante）。
